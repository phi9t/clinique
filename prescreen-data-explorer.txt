  What this plan does

  Goal: make the prescreen data layer correct and complete first — fetch all
  public data, parse it, and prove it conforms to our internal data model —
  before any LLM/decision stages.

  Core architectural move — convergence: four heterogeneous public sources
  collapse onto exactly two typed records:
  - Trial ← ClinicalTrials.gov (add search + pagination so we can pull a whole
  disease area, not a hand-typed NCT list)
  - PatientCorpus/PatientDocument ← Synthea (corpus-wide now), PMC-Patients
  (real free text → note docs), MIMIC-IV demo (structured
  labs/dx/meds/procedures)

  The source field is the discriminator, so validation is one contract, not 
  four.

  The conformance gate (validation.py) is the heart of the "maps to a correct
  model" ask:
  - controlled vocabularies (sex / phase / status / std_ages / doc source_types)
  - age-bound sanity (min ≤ max, non-negative, parse-failure detection)
  - duplicate doc_id detection, date parseability
  - the highest-value check: leakage — document.date > snapshot_date is an
  error, enforcing the design's as-of discipline

  Exposed as prescreen validate with a new meaningful exit code 7.

  Discipline preserved throughout: network fetch is split from pure parse
  (everything tested offline against tiny committed fixtures); runtime stays
  stdlib-only (urllib/json/csv/gzip); MIMIC-IV demo is real de-identified data
  so committed fixtures stay synthetic-shaped while the real fetch is documented
   in PROVENANCE.md.

  Explicitly out of scope: atomizer, retriever, judge, aggregator, evidence
  gate, orchestrator — the plan stops at "data fetched, parsed, and proven to
  conform."

     asserts that the

     The user wants to do the data layer first and do it right: fetch all the

     de-identified data → commit only synthetic-shaped fixtures; document the

  ★ Insight ─────────────────────────────────────
  - The fetch/parse split paid off across all four sources. Each new source has
  a network half (search_studies_raw, fetch_pmc_sample_raw, record_mimic) and a
  pure half (parse_search_page, parse_pmc_record, normalize_mimic). Only the
  pure halves are tested — so the entire 30-test prescreen suite runs offline in
   0.08s with zero network flakiness, exactly mirroring the existing
  fetch_study_raw-is-not-tested posture.
  - Controlled vocabularies as data, not code. Putting TRIAL_PHASES,
  PATIENT_SEX, etc. as frozensets in schemas.py (next to the records they
  constrain) means the model definition and its allowed values can't drift
  apart, and validation.py stays a thin consumer. An unexpected enum value now
  means "the API changed or we mis-mapped" — which is precisely the signal a
  conformance gate should raise.
  - The leakage check is the sharpest conformance test. It encodes a temporal
  invariant the type system can't: document.date ≤ snapshot_date. That's the
  difference between "the data is well-typed" and "the data behaves the way our
  mental model says it should."

  Overview

  Adds the "fetch all public data, parse, validate" layer to prescreen/: CT.gov
  search+pagination, three patient-source normalizers (Synthea corpus-wide,
  PMC-Patients, MIMIC-IV demo) converging on PatientCorpus, and a validation.py
  conformance gate wired to a prescreen validate CLI (exit 7). Synthetic
  fixtures + PROVENANCE for each source.

  Strengths

  - Fetch/parse split is held consistently across all sources — only pure halves
   are tested, so CI stays offline. This is the codebase's core discipline and
  it's respected everywhere.
  - Leakage check (document.date ≤ snapshot_date) is the standout: it encodes a
  temporal invariant the type system can't, directly serving the "conforms to
  our mental model" goal.
  - Controlled vocab as data in schemas.py keeps the model and its allowed
  values from drifting; validation.py stays a thin consumer.
  - PHI discipline correct: MIMIC-IV demo (real de-identified) gets
  synthetic-only fixtures with an explicit PROVENANCE warning; real fetch
  documented but not committed.
  - Determinism preserved (sorted ids, stable doc_ids), stdlib-only, exit-code
  convention extended cleanly (7).

  Issues & Risks

  1. prescreen search with no filters silently fetches the entire registry 
  (medium — real footgun).
  In search_studies_raw, if cond/term/status are all None, the query has only
  fields/pageSize/countTotal → it paginates all ~500k studies until
  nextPageToken runs out (no --max default either). One accidental clinique 
  prescreen search --out x.jsonl hammers the API. Suggest guarding in the CLI
  handler or the function:
  if not (cond or term or status) and max_studies is None:
      raise ValueError("refusing an unbounded search: pass 
  --cond/--term/--status or --max")

  2. Pagination/nctId edge cases (low).
  - record_search does
  payload["protocolSection"]["identificationModule"]["nctId"] — KeyError if a
  study lacks it. record_studies (existing) has the same shape, so it's
  consistent, but the search corpus is larger and more likely to hit an oddball
  record; a .get(...) chain with a clear error would be friendlier.
  - search_studies_raw loops while nextPageToken is truthy; a page with zero
  studies but a stale token would spin (won't happen with the real API, but the
  loop has no page-count ceiling). Minor since it's the untested network path.

  3. filter.overallStatus join separator unverified (low). You join statuses
  with ,. The v2 API has used both comma and pipe-style list encodings across
  revisions — worth a one-time live check before relying on multi-status
  filters, since this path isn't tested.

  4. Cross-source demographics shape is not uniform (low, but tension with the 
  "one model" goal).
  - Synthea emits {age, sex, birthdate}; PMC and MIMIC emit {age, sex} (no
  birthdate).
  - age semantics differ: Synthea computes age at snapshot_date; MIMIC uses
  anchor_age (age at anchor_year, not at snapshot). Both are reasonable given
  the sources, but a downstream consumer comparing demographics["age"] across
  sources is comparing slightly different things. Since demographics is an
  untyped dict, validation can't catch this. Worth a comment in mimic_demo
  noting the anchor_age semantics, and consider whether birthdate should be
  present (as None) everywhere for shape stability.

  5. validate with neither --trials nor --patients returns ok=True, exit 0 
  (low). A no-op invocation looks like a passing gate. Consider erroring (exit
  2) when both are absent.

  Nits:
  - record_mimic has a function-local import json; move to module top for
  consistency with the rest.
  - search_studies_raw always requests pageSize=100 even when max_studies < 100
  — one oversized fetch; cap page_size = min(page_size, max_studies) when set.
  - validate_patient_corpus age check isinstance(age, (int, float)) also accepts
   bool (bool ⊂ int); harmless here since age is never bool.

  Test Coverage

  Solid for the pure paths: search parsing + round-trip, PMC
  demographics/note/determinism, MIMIC domain mapping + gzip + corpus ordering,
  validation (clean fixtures pass, bad enum, min>max, leakage, on-snapshot
  allowed, bad source/sex).

  Gaps worth a test each: the age-bound parse-failure warning (age_unparsed),
  duplicate doc_id error, and observation-missing-value warning are implemented
  but unexercised. The leakage negative across a real corpus (MIMIC with
  snapshot before some doc) would be a nice end-to-end assertion.

  Well-structured, conventions-respecting, and the conformance gate genuinely
  delivers the requested guarantee. Issue #1 (unbounded search) is the only one 
  I'd fix before committing — it's a real operational footgun. The rest are
  low-severity polish. Want me to apply the search guard and add the three
  missing validation tests?
